Go HTML Templates: Functions and Flow

Templates are executed by applying them to a data structure
- [text/template docs](http://golang.org/pkg/text/template/)

The last post focused on how to structure Go templates. It left out the one essential feature of template systems: varying output. Let's look at my header.html:

<html>
<head><title>{{ . }}</title></head>
<body>

The {{ . }} means "copy the default representation of . to the output here." So, if you apply the template to the string "Index", it would end up with <title>Index</title>. The same goes for structs -- if you apply a template to a struct{Title string}, you can write the title with the action {{ .Title }}.

There's one catch with using ., and the solution was deep enough in the docs that I nearly missed it. Here's something that looks right, but isn't:

type Folder struct {
    Name string
    Files []string
}

f := Folder{"/root", []string{".bash_history", ".bashrc"}}

// It's not okay to ignore errors, unless you're a blog post.
tmpl, _ := template.New("t").Parse(
    `Contents of {{ .Name }}:
    {{ range _, $f := .Files }}
        {{ .Name }}/{{ $f }}
    {{ end }}
    `)
    tmpl.Execute(os.Stdout, sweaters)

Run this on play.golang.org.

This code fails to parse, because <.Name>: can't evaluate field Name in type string. The docs point out that range sets . to the current element. The missing piece is a few pages lower: $ holds the original value of .. Here's the correct template (or on play.golang.org):

template.New("t").Parse(
    `Contents of {{ .Name }}:
    {{ range .Files }}
        {{ $.Name }}/{{ . }}
    {{ end }}
    `)
    tmpl.Execute(os.Stdout, sweaters)

There's another context in which . changes -- when you invoke another template. In last post's index.html, you may have seen the line {{ template "header.html" "Index" }}. You can pass values from template to template, just like when invoking a template from Go. For a small enough project, that works fine. For a larger project, I expect to pass a larger struct into my header and footer, to dynamically include CSS and JavaScript files that are tailored to the page, rather than a static list in the header.

I've had to change my approach to writing HTML compared to working in Jinja2. Things I'd ordinarily do in the template are pushed out into my Go code, and it's starting to grow on me. I'll talk a bit about functions and control flow in templates next time, before moving on to a new topic.

With Jinja2 templates, you think in terms of a base structure that you override in pieces. Go templates feel less easy to extend from inside the templates, but you can do a lot from the code. You should probably read the text/template docs if you haven't already -- but, the gist is: you apply a template to a data structure, which is referred to by the . character. header.html expects to be applied to a string, which is used as the page title. In index.html, we pass the title as an argument into the {{ template }}.

This is one way in which Go's built-in API influences my design choices; with Jinja, I could get away with template files declaring their JavaScript includes in {% block %} overrides. In Go, I'm more likely to create a HeaderContext and FooterContext struct, and declaring the list of script files in my .go code.

A lot of Go's standard library feels like it's focused on the simplest thing that will do the job. So, while I miss the flexibility of Jinja2, Go templates can do everything I need -- I'm just not used to it. A pattern that's beginning to emerge from the Go standard library: functional, production-friendly, but inconvenient.

I've talked a lot about the syntax of templates here -- the next post will spend more time on how I actually use the http/template package in my code.
